1. Introduction 
1.1 Purpose 
This document specifies the functional and non-functional requirements for CodeVault, a self-hosted code snippet management platform. It serves as the engineering reference during implementation and as a design record demonstrating full-stack architecture, database design, and authentication engineering for portfolio and internship review. 
1.2 Scope 
CodeVault allows a developer to store, tag, search, and organize reusable code snippets, fork snippets from other users, view usage analytics, and maintain a public profile showcasing their snippet library. A single AI-assisted capability — natural-language explanation of a snippet's behavior — is included as a secondary feature; it is not the product's core identity. 
Version 1 (this document) targets a single-server, single-region deployment suitable for individual use and portfolio demonstration. Team workspaces and organization-level sharing are explicitly out of scope for v1 (see Section 10). 
1.3 Intended Audience 
Solo developer (author) — implementation reference 
Internship/hackathon reviewers — architecture and design record 
Future contributors — onboarding reference if open-sourced

2. Overall Description 
2.1 Problem Statement 
Developers accumulate reusable code — utility functions, configuration templates, regex patterns, boilerplate — across projects, but rarely have a structured place to store it. GitHub Gists provide storage without organization: no tagging, no search by language, no analytics, and no lightweight way to browse a personal library. The result is snippets scattered across notes apps, Slack messages, and forgotten repositories. 
2.2 Proposed Solution 
CodeVault provides a structured, searchable, self-hostable home for code snippets. Every snippet carries a language, a set of tags, and visibility settings. Snippets can be forked the way repositories are forked on GitHub, preserving a lineage back to the original author. A public profile page lets a developer showcase their snippet library the way a portfolio site showcases projects. 
2.3 Product Perspective 
CodeVault is a standalone full-stack web application: a Next.js frontend, an Express REST API, and a PostgreSQL database accessed through Prisma. It has no dependency on any external platform and is designed to be self-hosted via Docker by any individual developer. 
2.4 Product Functions — Summary 
User authentication (signup, login, JWT-based session management) 
Snippet CRUD with language, title, description, and tags 
Full-text and tag-based snippet search 
Snippet forking with lineage tracking 
View-count analytics per snippet 
Public, shareable user profile pages 
AI-assisted snippet explanation (single feature, on-demand) 
2.5 User Classes and Characteristics 
User Class 
Characteristics & Needs 
Registered User 
Creates, edits, tags, and organizes their own snippets. Can fork public snippets from others. 
Visitor (unauthenticated) 
Can browse public snippets and public profiles, but cannot create or fork without an account. 
2.6 Operating Environment 
A Dockerized deployment (application container + PostgreSQL container) runnable on any machine with Docker installed, or deployable to a platform such as Railway or Render for public hosting. The frontend is a standard responsive web application accessible from any modern browser. 
2.7 Design and Implementation Constraints 
PostgreSQL only — no additional datastore dependency for v1 
Authentication must be self-contained (JWT) — no dependency on a third-party auth provider for v1 
The AI explanation feature must degrade gracefully — the rest of the product must work fully if the AI API is unavailable 
2.8 Assumptions and Dependencies 
Single-server deployment is sufficient for v1 — no horizontal scaling requirement at this stage 
The AI explanation feature depends on an external LLM API (Anthropic Claude) and is treated as an optional enhancement, not a core dependency

4. External Interface Requirements 
4.1 User Interfaces 
A responsive Next.js web application covering: signup/login, snippet creation and editing, a searchable snippet library view, an individual snippet detail view with syntax highlighting, and a public profile page. 
4.2 Hardware Interfaces 
None. Standard client hardware — a modern browser and an internet connection — is sufficient. 
4.3 Software Interfaces 
Anthropic Claude API — used for the on-demand snippet explanation feature only 
PostgreSQL — primary and only datastore 
4.4 Communication Interfaces 
The frontend communicates with the backend over HTTPS via a REST API secured with JWT bearer tokens. The AI explanation feature makes a server-side call from the Express API to the Anthropic API — the frontend never calls the AI provider directly, which keeps the API key server-side and lets the backend apply rate limiting to that specific route.

6. System Architecture 
6.1 High-Level Architecture 
CodeVault follows a conventional three-tier architecture with one clearly isolated external dependency. The Next.js frontend never talks to PostgreSQL or the AI provider directly — every request flows through the Express API, which enforces authentication, ownership checks, and validation before touching the database or calling out to the AI service.

6.4 Data Flow — Example: Creating and Forking a Snippet 
1. A user submits a new snippet through the dashboard. 2. The API validates the JWT and the request body, then the Snippet Service writes the record via Prisma. 3. Another user browses public snippets and selects Fork. 4. The Fork Service creates a new snippet record owned by the forking user, storing a reference to the original snippet's ID. 5. The original snippet's fork count is incremented. 6. Both the original and the fork are independently editable from that point forward.

7.2 Entity Relationships — Narrative Overview 
A User has many Snippets; each Snippet belongs to exactly one owning User. 
A Snippet may optionally reference a parent Snippet via forked_from_id, forming a self-referential fork lineage. 
Tags are many-to-many with Snippets through the SnippetTag join table, allowing search and filtering by tag without duplication. 
SnippetView records are appended on each public view, enabling both a simple counter and, later, time-based view analytics without a schema change. 
8. Use Cases 
UC-1 — Register and Create First Snippet 
Actor 
Visitor 
Description 
A new user signs up and creates their first snippet. 
Preconditions 
None. 
Main Flow 
1) Visitor registers with email/password. 2) System hashes password and creates the User record. 3) System issues a JWT. 4) User creates a snippet with title, code, and language. 
Postcondition 
A new User and their first Snippet exist in the database. 
UC-2 — Search and Filter Snippets 
Actor 
Registered User or Visitor 
Description 
A user searches their own library or browses public snippets by tag/language. 
Preconditions 
At least one public snippet exists in the system for a visitor; none required for a user browsing their own library. 
Main Flow 
1) User enters a keyword or selects a tag/language filter. 2) System queries matching snippets. 3) Results are displayed with syntax-highlighted previews. 
Postcondition 
None — read-only operation. 
UC-3 — Fork a Public Snippet 
Actor 
Registered User 
Description 
A user copies another user's public snippet into their own account. 
Preconditions 
The target snippet is marked public; the acting user is authenticated. 
Main Flow 
1) User views a public snippet. 2) User selects Fork. 3) System creates a new Snippet owned by the acting user, referencing the original via forked_from_id. 4) The original snippet's fork_count is incremented. 
Postcondition 
A new, independently editable Snippet exists, with lineage preserved to the original. 
UC-4 — Request AI Explanation 
Actor 
Registered User 
Description 
A user requests a natural-language explanation of one of their snippets. 
Preconditions 
The user owns or is viewing the snippet; the AI service is reachable. 
Main Flow 
1) User selects Explain on a snippet. 2) API sends the snippet body to the Anthropic API. 3) Response is streamed or returned to the frontend and displayed below the snippet. 
Postcondition 
No data is persisted from this action in v1 — the explanation is generated on demand each time. 
UC-5 — View a Public Profile 
Actor 
Visitor or Registered User 
Description 
Anyone views another user's public profile and their public snippet library. 
Preconditions 
The profile owner has at least one public snippet, or an empty state is shown. 
Main Flow 
1) Visitor navigates to a user's public profile URL. 2) System returns the user's display name, bio, and list of public snippets. 
Postcondition 
None — read-only operation.

10. Non-Goals (v1) 
Explicitly excluded from v1 to keep the project achievable on a solo, time-boxed timeline. These are natural v2 candidates once the core product is deployed and documented. 
Team workspaces or shared/organization-owned snippet libraries 
Snippet versioning or edit history beyond the current state 
Real-time collaborative editing 
Comments or social interaction features beyond forking 
Persisted AI explanations (v1 generates on-demand only, nothing is cached or stored)
